◆Puiky · Nota de arquitectura

Anatomía de un agente que fluye

Cómo está construido Puiky —un asistente personal por voz y texto— y, sobre todo, qué decisiones de diseño hacen que se sienta como hablarle a alguien, y no como llenar un formulario.

La fluidez no viene del modelo. Viene de cómo se le da el contexto, cómo se resuelven las referencias, y cómo se manejan el error y la memoria. Diez decisiones, no un modelo más grande.

El recorrido

De una frase a una acción

Ejemplo real: "gasté 20 mil en mercado con efectivo"

  1. El bot recibe el mensaje (si es voz, Whisper lo transcribe primero) y llama a interpret(texto, historial).
  2. Se arma el prompt: sistema + contexto (fecha, categorías, cuentas…) + memoria de la conversación + el mensaje + las herramientas disponibles.
  3. El modelo decide: emite un tool call → registrar_gasto(20000, "mercado", "efectivo").
  4. Se ejecuta: la tool resuelve "mercado" → categoría Comida y "efectivo" → la cuenta, y registra el gasto (actualiza el saldo).
  5. Segunda pasada, sin tools: se le devuelve el resultado real para que redacte la confirmación.
  6. Responde: "Registré 20.000 en Comida. Tu saldo es 80.000." Si fuera un borrado, aparecen botones Sí / Cancelar.
El núcleo · lo que hay que aprender

Diez decisiones que dan fluidez

Un agente funciona con solo exponer funciones. Se siente fluido por esto. Cada punto trae el anti‑patrón que evita.

1

Tools = intenciones, no un CRUD genérico

Cada herramienta es un verbo con significado (registrar_gasto, anadir_a_hoja, cumplir_responsabilidad), no insertar_fila(tabla, datos). El modelo elige mucho mejor entre verbos claros, y cada tool encapsula sus reglas.

Anti‑patrón: exponer la base cruda o un único "ejecutar_sql". El modelo se pierde y hace cosas peligrosas.

2

Referencias por nombre, nunca por ID

El usuario dice "la tarea del informe", "la cuenta de ahorros". Las tools resuelven eso a un id internamente (por coincidencia de nombre o por búsqueda semántica), y si hay ambigüedad preguntan cuál.

Anti‑patrón: que la tool pida un UUID. Nadie los tiene a mano; el modelo los inventa o falla. Es lo que más rompe un agente.

3

Inyecta el contexto en el prompt

Antes de cada interpretación, el prompt incluye la fecha/hora y las listas de lo que existe: cuadernos, portafolios, categorías, cuentas, proyectos. Así mapea "mercado"→Comida y "mañana"→fecha ISO sin preguntar.

Anti‑patrón: un prompt estático. El modelo no sabe qué existe, así que pregunta todo o inventa nombres.

4

Dale memoria de conversación

El bot guarda los últimos turnos por chat y los envía en cada petición. Cuando el agente pregunta "¿cuál de los dos ítems?", la respuesta del usuario tiene contexto.

Anti‑patrón: agente sin estado. Cada mensaje se interpreta de cero; "bórralo" se rutea a la cosa equivocada. Mata cualquier diálogo de dos pasos.

5

Los errores son datos, no excepciones

Si una tool falla, devuelve { ok: false, error: … } en vez de reventar. El modelo lee ese error y lo explica ("no encontré esa categoría, ¿la creo?") o pide el dato que falta.

Anti‑patrón: dejar que una excepción mate el turno y devuelva un "error 500" al usuario.

6

Confirma lo destructivo fuera del modelo

Las tools de borrado no borran: piden confirmación. El bot muestra botones Sí / Cancelar; solo al tocar "Sí" se borra, vía la API — no vía el modelo.

Anti‑patrón: que un LLM borre datos por su cuenta a partir de una frase ambigua.

7

Abstrae el modelo (real / fake)

El proveedor de LLM es una interfaz con dos implementaciones: real (Qwen vía Ollama) y fake (determinista, por reglas). Igual para voz y embeddings. Se desarrolla y testea sin el modelo, y se cambia con una variable de entorno.

Anti‑patrón: acoplar todo el código a un SDK o modelo concreto.

8

Una ronda de tools + respuesta con datos reales

El orquestador pide qué hacer, ejecuta las tools (varias, si hay multi‑intención) y hace una segunda llamada sin tools solo para redactar con los resultados reales (saldos, ids), no con lo que el modelo "cree".

Anti‑patrón: dejar al modelo iterar sin límite, o narrar sin ver lo que de verdad pasó.

9

No preguntes lo que ya se dijo

Se le instruye explícitamente a deducir del contexto y no re‑preguntar datos ya presentes. Combinado con la memoria (4) y el contexto (3), casi nunca pide algo dos veces.

Anti‑patrón: un interrogatorio: "¿de qué cuenta?" cuando el usuario ya dijo "con efectivo".

10

Responde breve y confirma lo hecho

Respuestas cortas, en el idioma del usuario, que confirman la acción concreta ("Registré…", "Listo, moví…"). El tono es de asistente, no de sistema.

Anti‑patrón: párrafos genéricos que no dicen si algo pasó de verdad.

Para replicar

La receta, en una lista

  • Verbos, no CRUD. Define tools por intención del usuario.
  • Resuelve referencias por nombre o significado. Nunca pidas IDs.
  • Desambigua: si hay varios candidatos, pregunta cuál.
  • Inyecta contexto — la fecha y los nombres de sus cosas — en el prompt.
  • Da memoria de conversación, aunque sean pocos turnos.
  • Errores como datos, no como excepciones.
  • Confirma lo destructivo fuera del modelo.
  • Una ronda de tools + respuesta con resultados reales.
  • Abstrae el modelo con una interfaz y un doble "fake".
  • Sé breve y no preguntes lo que ya se dijo.

En resumen

Por qué "el otro agente" no fluye igual

Cuando un agente parecido se siente rígido, casi siempre falta uno de estos: pide IDs o datos que el usuario no tiene (2, 3); es stateless y pierde el hilo (4); sus tools son genéricas o mal descritas (1); revienta ante un fallo (5); o el prompt no le dice qué existe (3).

La inteligencia estaba disponible todo el tiempo. Lo que faltaba era darle el mundo del usuario: sus nombres, su fecha, su conversación, y una forma segura de actuar.
